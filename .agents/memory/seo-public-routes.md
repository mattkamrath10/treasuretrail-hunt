---
name: SEO public routes + share-path gating
description: How public SEO landing routes coexist with UUID detail routes, and the segment-boundary rule for the guest share-path gate.
---

- Public SEO landing routes (county `/ca/:county`, city, city-category, `/category/:slug`, `/seller/:handle`, plus slug variants of `/wanted/:id` and `/event/:id`) render via `PublicSeoRoute` + `PublicSeoPage`. UUID vs slug is decided by a UUID regex hybrid wrapper in `AppShell` — UUID -> real detail page, slug -> SEO page. Never collapse the two; both must keep working.
- Static routes (`/seller/new`) outrank dynamic (`/seller/:handle`) in React Router v6, so order doesn't matter, but **the guest share-path gate in `App.tsx` must match on whole path segments** (`pathname === p || pathname.startsWith(p + '/')`), NOT `startsWith(p)`.
  - **Why:** a prefix check wrongly gated public handles like `/seller/new-deals` (starts with `/seller/new`) behind auth/onboarding.
  - **How to apply:** any time you add a private `/seller/*` (or similar) prefix, keep the segment-boundary matcher; drop trailing slashes from the prefix list.
- The SEO read migration opens `profiles`/`marketplace_listings`(active)/`events`(published) SELECT to `anon`. `profiles` has NO PII (email/phone live in `auth.users`); its columns are all public-seller-display (username, bio, avatar, business_*, founding_partner, membership_tier), so `USING(true)` is acceptable for storefront SEO. Re-check this if a sensitive column is ever added to `profiles`.
- Sitemap is generated by a `prebuild` step (`scripts/generate-sitemap.mjs`) that reads `routeCatalog.json` statically and tries a Supabase listing fetch that **soft-fails to `[]`** — so Codemagic/no-secret builds still succeed.
