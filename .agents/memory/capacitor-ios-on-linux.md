# `npx cap add ios` succeeds on Replit (no Mac needed for scaffolding)

Two gotchas, both surprising:

1. **Capacitor 8's CLI hard-requires Node >= 22.** On Node 20 it aborts with
   `[fatal] The Capacitor CLI requires NodeJS >=22.0.0`. Fix: install the
   `nodejs-22` module via package management (it becomes the project runtime;
   verified Vite + Express build/run fine on it).

2. **`cap add ios` does NOT need CocoaPods / a Mac to scaffold.** Capacitor 8
   defaults to **Swift Package Manager** (`ios/App/CapApp-SPM/Package.swift`)
   instead of CocoaPods, so there is no `pod install` step — the whole `ios/`
   Xcode project generates cleanly on Linux. Plugins (e.g.
   `@capacitor-firebase/messaging`) are auto-registered into Package.swift and
   `capacitor.config.json` (packageClassList).

**Why it matters:** the user has no Mac. The `ios/` folder can be generated +
committed here, then archived/uploaded from a cloud-Mac CI. Only the final
archive/sign/upload truly needs macOS + Xcode.

**How to apply:** `npx cap add ios` then `npx cap sync ios` are safe on Replit.
Generated `ios/App/App/capacitor.config.json` + `config.xml` are gitignored by
Capacitor's own `ios/.gitignore` (regenerated by sync) — that's expected, not a
mistake.

## Changing the appId/bundle ID after `cap add` — sync does NOT do it all

`npx cap sync ios` regenerates the gitignored `ios/App/App/capacitor.config.json`
(and `config.xml`) from `capacitor.config.ts`, so the appId there updates
automatically. BUT it does **not** rewrite `PRODUCT_BUNDLE_IDENTIFIER` in
`ios/App/App.xcodeproj/project.pbxproj` — Capacitor only writes that at `cap add`
time. To rename the bundle ID you must hand-edit both pbxproj occurrences
(Debug + Release build configs) and then run sync. Verify with
`rg "com\.old\.id" ios` returning nothing. (`config.xml` in Capacitor 8 has no
`id` attribute — it's a near-empty Cordova stub, not a place the appId lives.)
